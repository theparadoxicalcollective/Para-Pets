SECTION 1 — WHAT THE GAME IS
================================================================================

ParaPets is a browser-based fantasy RPG set in a world called Veridia.
Players hatch and raise magical creatures called ParaPets, battle enemies
across themed regions, go fishing with a real-time minigame, trade on a
player marketplace (the Mire Bazaar), and decorate a personal Pet House.

The tech stack is:
  Frontend  — React + TypeScript, Vite, Tailwind CSS, Wouter (routing),
               TanStack Query v5, Framer Motion, shadcn/ui, Lucide React
  Backend   — Node.js + Express + TypeScript, Drizzle ORM, Passport.js
  Database  — PostgreSQL (via Drizzle)
  Payments  — Stripe (client: server/stripeClient.ts, webhooks: server/webhookHandlers.ts)

The app is full-stack in a single repo. The Express server serves both the
REST API and the Vite-compiled frontend on the same port.

SECTION 2 — CORE GAMEPLAY LOOP
================================================================================

1. EARN COINS
   - Catch fish and sell them, or defeat enemies to collect coin drops.

2. BUY EGGS / GEAR
   - Spend coins at World Shops (region-specific) to buy pet eggs, fishing
     poles, bait, potions, accessories, and power-up consumables.

3. HATCH & RAISE
   - Eggs hatch over real time (hours). Use Hatch Accelerator items to speed
     this up. Once hatched, feed pets power-up items to raise their stats.

4. BATTLE
   - Take pets into region encounters. Slash enemies with swipe gestures, parry
     attacks, and fire special skills. Defeated enemies drop coins, XP, and
     rare items.

5. FISH
   - Equip a pole and optional bait. Complete the tension/reel minigame to catch
     fish for the Fish Book (collection) or sell them for coins.

6. SPECIALIZE
   - Better fishing gear → rarer fish → more coins.
   - Better battle gear / higher-level pets → tougher enemies → rarer drops.

7. ENDGAME
   - Complete the Fish Book, climb the PvP Arena rankings, unlock all Veridia
     regions, and fully outfit pets with accessories.

SECTION 3 — FISHING SYSTEM OVERVIEW
================================================================================

FILES:
  client/src/pages/FishingPage.tsx   — all minigame UI & logic
  server/routes.ts                   — cast, catch, server-side validation
  shared/schema.ts                   — shopItems (fish), playerFishInventory,
                                       playerFishingEquipment

--- 3.1 Equipment ---

POLES
  - Stored in user_inventory; linked to a shop_item with type "fishing" and
    fishingType "pole".
  - poleMaxUses: total durability. poleUsesLeft: remaining uses.
  - Every cast attempt (successful or not) decrements poleUsesLeft.
  - High-tier poles (3★-5★) have poleSlowdown3/4/5 multipliers that reduce
    how fast line tension rises, making harder fish manageable.
  - Some poles have rareCatchBoostPercent which increases the weight of 4★
    and 5★ fish in the selection pool.

BAIT
  - Consumable. One unit consumed only on a SUCCESSFUL catch.
  - baitCatchBoost: increases the progress gain rate during reeling.
  - baitRarityBoostStar + rarityBoostPercent: boosts the weight of a specific
    rarity tier in the pond's fish pool.

--- 3.2 Fishing Phases ---

  IDLE       → Player selects pole and bait.
  CASTING    → Animation; verifies pole is equipped and has durability.
  WAITING    → Random delay 1.5s–4s; server selects the fish from the pond.
  NIBBLE     → Player must tap/click within the window (shorter for rarer fish).
                 5★ fish: 1.1 s window, 1 chance.
                 1★ fish: 2.5 s window, 3 chances.
  REELING    → Core tension/reel minigame (see 3.3).
  CAUGHT/MISSED → Final outcome.

--- 3.3 Tension & Reel Minigame ---

Three simultaneous meters:

  CATCH PROGRESS (0–100%): Fills while player holds "Pull". Reach 100% → fish caught.
  TENSION       (0–100%): Rises while holding "Pull", falls when released.
                           Sweet spot is roughly 35%–65% for 5★ fish.
                           Hitting 100% → line snaps → fish lost.
  ESCAPE METER  (0–100%): Fills when player releases OR is outside sweet spot.
                           Hitting 100% → fish escapes.

Fish cycle through four behaviour states during the reel phase:
  CALM   — standard rates.
  RESIST — tension rises 50% faster, progress 30% slower.
  TIRED  — tension rises 50% slower, progress 50% faster (player's advantage).
  SURGE  — tension spikes at 2.2× rate, player MUST release to avoid snapping.

--- 3.4 Server-Side Catch Logic ---

  Base rarity weights: 1★:60  2★:24  3★:10  4★:4  5★:2
  Pole and bait modifiers adjust these weights before selection.
  Client sends a performanceScore to the server after the minigame.
  If performanceScore = 100 → guaranteed catch.
  If lower → catch chance = 0.4 + (score / 100) * 0.5

SECTION 4 — BATTLE / SLASH SYSTEM OVERVIEW
================================================================================

FILES:
  client/src/components/BattleArena.tsx  — PvE combat engine
  client/src/pages/PvpBattlePage.tsx     — PvP combat engine
  server/routes.ts                       — /api/explore/:locationId/encounter,
                                           /api/explore/defeat/:enemyId
  shared/schema.ts                       — locationEnemies, enemyDrops

--- 4.1 Slash / Hit Mechanics ---

  "Fruit Ninja"-style continuous swipe detection (not tap-to-hit).
  - onPointerDown starts recording a slashPath (up to 22 coordinate points).
  - Each frame: checkSegmentHit uses segmentPointDist to test whether the
    latest swipe segment intersects an enemy's circular hitbox.
  - hitEnemiesRef (Set) prevents a single continuous motion from hitting the
    same enemy more than once per frame.
  - Combo System: Successive hits within 1.4 s increase comboCount.
    Damage multiplier = 1 + min(combo × 0.12, 0.6).

--- 4.2 Parry and Counter System ---

  Enemies "charge" an attack (red ring + progress bar) for ~1100–1400 ms.
  At 320 ms into the charge, parryWindowOpen = true.
  Player sees a "⚠ ATTACK!" warning and a red ring on their pet.

  SUCCESS PARRY — player taps their pet during the parry window:
    → Incoming damage nullified.
    → "BLOCK!" animation plays.
    → Counter State activates: next 3 hits deal 2× damage.

  FAILED PARRY — charge completes before player taps:
    → Pet takes damage reduced by its DEF stat only.

--- 4.3 Mana and Special Skills ---

  Mana fills passively over time, +14 on being hit, +13–22 on landing slashes
  (higher with combos/crits).
  At 100 mana the pet's aura shifts to its rarity colour.
  Tapping the pet fires fireSkill().

  Skill types:
    LAZER / POISON   — High single-target damage or damage-over-time.
    BUBBLE           — AoE damage to all enemies on screen.
    HEAL SELF/PARTY  — Restores HP based on ATK × skillHealPercent.

--- 4.4 Enemy Waves and Boss Encounters ---

  Encounters are fetched from /api/explore/:locationId/encounter.
  BattleArena iterates through the returned enemy array via waveIndex,
  showing a "Wave Complete" phase between each wave.

  BOSSES:
    - Larger hitbox (radius 24 vs 17 for normal enemies).
    - Rage State triggers at ≤50% HP: erratic movement, red screen tint,
      reduced charge time (1100 ms → 980 ms).
    - Hits cause screen shake and shockwave ring effects.

--- 4.5 PvP Mechanics ---

  AI pets wander toward the nearest player pet.
  Damage comes from collision (not from parry/charge).
  Skill targeting requires tapping the specific opponent pet (e.g., Lazer).

--- 4.6 Rewards ---

  On enemy defeat → POST /api/explore/defeat/:enemyId
  Server calculates XP (added to active pet's levelPoints), coins, and
  possible item drops defined in the enemyDrops table.

SECTION 5 — PET SYSTEM OVERVIEW
================================================================================

FILES:
  shared/schema.ts                        — shopItems, userInventory,
                                            petEquippedAccessories,
                                            petHousePositions, petTemplates,
                                            petTemplateParts
  server/routes.ts                        — all /api/pet/... endpoints
  client/src/components/PetDetailPage.tsx
  client/src/components/PetAnimator.tsx
  client/src/pages/PetHousePage.tsx
  client/src/pages/PetWorldPage.tsx

--- 5.1 Pet Stats ---

  Stored in user_inventory per owned pet.
  petHealth  — base 1000
  petAtk     — base 50
  petDef     — base 50
  petLevel   — 1 to 100
  petLevelPoints — XP toward next level
  Rarity (1–5★) — defined in shop_items; controls power-up limits.

--- 5.2 Hatching ---

  Purchase sets hatchStartedAt to current time.
  hatchTime (hours) is defined on the shop_item.
  GET /api/pet/:inventoryId/hatch-check compares elapsed time to hatchTime.
  If complete → isHatched = true.
  Special items with specialType "hatch_time" shift hatchStartedAt backward
  to accelerate hatching.

--- 5.3 Leveling and Power-Ups ---

  XP formula: 100 + level × 30 + level² × 5

  Power-up slots per level = (Rarity + 2) × Level
  Once that many items have been used this level, no more stat items until
  the next level-up (resets itemsUsedThisLevel counter).

  Stat-boosting items target: atk, def, health, or lvl (direct XP grant).
  POST /api/pet/:inventoryId/power-up

  Reset for 300 coins → Level 1, base stats, accessories removed.
  POST /api/pet/:inventoryId/reset-stats

--- 5.4 Accessories ---

  Pets have 3 equip slots.
  POST /api/pet/:inventoryId/equip — adds atkBoost, defBoost, or healthBoost
    from the accessory's shop_item definition.
  POST /api/pet/:inventoryId/unequip — reverses those bonuses.
  Tracked in pet_equipped_accessories (junction table).

  Nicknames (up to 20 chars) — PATCH /api/inventory/:inventoryId/nickname.

--- 5.5 Pet Templates and Visuals ---

  pet_templates + pet_template_parts tables.
  Each template is assembled from named parts (Body, Eyes, Mouth, Wings, Tail)
  with posX, posY, zIndex offsets.
  PetAnimator.tsx renders idle/hover states.
  In PetWorldPage: canFly = true → floating animation; false → ground drift.

--- 5.6 Pet House ---

  Players drag-and-drop pets within their house.
  Positions (posLeft, posTop) saved to pet_house_positions.
  PATCH /api/pet-house-positions/:inventoryId
  Other players can visit to view the collection.

SECTION 6 — ITEMS AND INVENTORY
================================================================================

FILES:
  shared/schema.ts      — shopItems, userInventory, playerFishingEquipment,
                          playerFishInventory, petEquippedAccessories,
                          playerMarketListings
  server/routes.ts      — all item/inventory endpoints
  client/src/components/PetInventory.tsx
  client/src/components/PetDetailPage.tsx

--- 6.1 Item Types (shop_items.type field) ---

  pet          — Eggs / hatched pets.
  power_up     — Consumable stat boosters (atk, def, health, lvl).
  item         — General consumable or resource.
  special      — Special effect (e.g., hatch accelerator).
  accessory    — Equippable passive gear for pets.
  fishing      — Fishing gear (sub-typed by fishingType: pole | bait | fish).
  potion       — In-battle consumables (Healing, Mana, Revive).

--- 6.2 Inventory Tables ---

  user_inventory          — Core ownership table: userId, itemId, quantity,
                            all pet stats (health, atk, def, level, xp),
                            isHatched, hatchStartedAt, nickname, marketListed.
  player_fishing_equipment — Tracks currently active pole and bait per user.
  player_fish_inventory   — Records caught fish per player.
  pet_equipped_accessories — Junction: links accessory inventoryId → pet inventoryId.
  player_market_listings  — Active listings on the Mire Bazaar.

--- 6.3 Key Inventory Rules ---

  - Pole durability decreases every cast (not just on success).
  - Bait is consumed only on a successful catch.
  - Accessories cannot be equipped if all 3 slots are full; the player must
    unequip one first.
  - Potions are used during battle from a dedicated hotbar (not the main UI).
  - A pet can be listed on the market while still in the player's house; it
    becomes unavailable for battle while listed.

SECTION 8 — HOW FISH AND ITEMS ARE ADDED FROM THE ADMIN PANEL
================================================================================

--- 8.1 Adding a Regular Item ---

  1. Open Admin → Item DB section.
  2. Choose item type (power_up, accessory, potion, special, etc.).
  3. Fill in: name, description, price, rarity (1–5), stat boosts.
  4. Upload image → server processes it with Sharp for consistent sizing.
  5. Item appears in the global shop_items database.
  6. To make it available in a world region: assign it via the Map/World editor.
     Route: POST /api/admin/location/:locationId/assign-item/:itemId

--- 8.2 Adding a Pet ---

  STEP 1 — Build the Pet Template (Admin → Animation Parts):
    a. Create a new PetTemplate record.
    b. Upload individual body parts (eyes, body, tail, wings, etc.) for both
       front and back views, with posX/posY/zIndex for each.
    c. Trigger assembly: POST /api/admin/pet-templates/:id/assemble
       This composites parts into a preview image.

  STEP 2 — Create the Pet Item (Admin → Item DB → Pets):
    a. Create a new item with type "pet".
    b. Link it to the PetTemplate created in Step 1.
    c. Upload Egg image and Hatched image.
    d. Set hatchTime (hours), base stats, rarity.
    e. Assign to a world shop as in 8.1 step 6.

--- 8.3 Adding a Fish ---

  1. Open Admin → Fishing section (FishingAdminPanel.tsx).
  2. Create a new item with type "fishing" and fishingType "fish".
  3. Set: name, rarity, sell price, weight in the pool.
  4. Upload Fish Parts (Body, Eyes, Tail) individually via
     POST /api/admin/fish-parts/:fishItemId
  5. Position each part on the coordinate system via
     PATCH /api/admin/fish-parts/:partId
     The minigame uses these coordinates to render the fish dynamically.
  6. Assign the fish to a fishing location/pond so it appears in catches.

--- 8.4 Adding a Fishing Pole or Bait ---

  Same flow as a regular item (8.1) but:
  - type = "fishing", fishingType = "pole" or "bait".
  - Set pole-specific fields: poleMaxUses, poleSlowdown3/4/5,
    rareCatchBoostPercent.
  - Set bait-specific fields: baitCatchBoost, baitRarityBoostStar,
    rarityBoostPercent.

SECTION 10 — IMPORTANT GAME MECHANICS SUMMARY
================================================================================

COINS
  - Primary currency. Earned from fishing sales, enemy drops, and market sales.
  - Stored as integer on the user record.

RARITY SYSTEM (1–5★)
  - Applies to fish, pets, accessories, and drops.
  - Higher rarity → better stats, harder to obtain, more power-up slots per level.

WORLD / LOCATION SYSTEM
  - Veridia is divided into themed regions (Swamp, Volcanic, Sky Realm, etc.).
  - Each location has its own: shop items, enemies, fishing pond, and NPCs.
  - The admin assigns items and enemies to specific locations.

MARKET (MIRE BAZAAR)
  - Fully player-to-player. Items listed in player_market_listings.
  - Pets listed on market are unavailable for battle until unlisted.

AQUARIUM
  - Players can display caught fish in a personal aquarium instead of selling.

SUPPORT SYSTEM
  - Players can submit support tickets from within the game.
  - Admin panel has an inbox to view and respond.

SOUND SYSTEM
  - lib/sounds.ts manages all in-game audio.
  - Sounds are triggered at key game events (catch, battle hit, level-up, etc.).

AUTHENTICATION
  - Passport.js handles sessions.
  - User record stores: username, password hash, coins, isAdmin flag, activePetId.

STRIPE PAYMENTS
  - Used for premium currency or shop purchases.
  - Webhooks handled in server/webhookHandlers.ts.

SECTION 11 — WHAT AN AI SHOULD KNOW BEFORE EDITING CODE
================================================================================

1. SCHEMA IS THE SOURCE OF TRUTH
   shared/schema.ts defines every database table, column, and type.
   Always check this file first before adding new fields or endpoints.
   Use createInsertSchema from drizzle-zod to derive insert types.

2. STORAGE LAYER
   All database reads/writes go through server/storage.ts (IStorage interface).
   Do NOT write raw SQL in routes.ts — add a method to IStorage and implement
   it in the storage class, then call it from the route.

3. FRONTEND DATA FETCHING
   Uses TanStack Query v5 (object form only, no positional args).
   Default fetcher is already configured; queries just need a queryKey.
   Mutations use apiRequest from @lib/queryClient.
   Always invalidate affected queryKeys after mutations.

4. ROUTING
   Frontend: wouter (Link, useLocation). Never use window.location directly.
   Backend: all game routes in server/routes.ts, prefix /api/.
   Admin routes are protected by isAdmin middleware on the server AND
   conditionally rendered on the client.

5. DO NOT MODIFY THESE FILES without a very good reason:
   - vite.config.ts
   - drizzle.config.ts
   - package.json (use package manager tools to add dependencies instead)

6. ENVIRONMENT / IMPORTS
   - React is auto-imported by the JSX transformer. Do NOT import React manually.
   - Use import.meta.env.VITE_* for frontend env vars (not process.env).
   - Use @assets/... for game images in the frontend.
   - Use @/ prefix for src/ aliases (components, hooks, lib, pages).

7. FISHING MINIGAME STATE MACHINE
   The fishing page is a complex state machine. When editing it, trace the
   full phase flow (IDLE → CASTING → WAITING → NIBBLE → REELING → CAUGHT/MISSED)
   and ensure state transitions are not broken.

8. BATTLE SYSTEM IS FRAME-BASED
   BattleArena.tsx uses requestAnimationFrame loops. Be careful when adding
   state updates inside these loops — ensure you are using refs where needed
   to avoid stale closure bugs.

9. PET TEMPLATE ASSEMBLY
   Pet visuals depend on the pet_template_parts coordinate system. If you add
   new pets, they MUST be assembled through the admin panel, not hard-coded.

10. IMAGE UPLOADS
    The server uses Sharp to process uploaded images. Keep this pipeline intact
    when adding new upload endpoints.

11. ADDING NEW ITEMS OR FISH
    Always use the Admin Panel flow described in Section 8. Do not seed items
    directly in SQL without also updating the admin UI to manage them.

12. POWER-UP SLOT FORMULA
    power-up slots = (rarity + 2) × level
    This controls how many stat items a pet can consume per level. Any changes
    to rarity or leveling must re-check this formula.

13. PET LEVELING HELPERS (server/routes.ts)
    Two helper functions near the top of routes.ts centralise all leveling math:
      xpForLevel(level)  → XP needed to reach the NEXT level from `level`
      applyPetXp(currentLevel, currentPoints, pointsToAdd)
                         → { newLevel, newPoints }
    These are called by three endpoints: power-up, use-special, and feed-edible.
    Do NOT inline the leveling formula again — always call applyPetXp().

14. SERVER-SIDE CATCH VALIDATION
    The fishing minigame is client-side but the final catch decision is server-
    side. The client sends a performanceScore; the server does the final roll.
    Never bypass this by trusting only client state.

15. DATA-TESTID ATTRIBUTES
    All interactive and meaningful display elements should have data-testid
    attributes following the pattern: {action}-{target} or {type}-{content}.
    Dynamic lists: {type}-{description}-{id}.

SECTION 12 — DATABASE STRUCTURE
================================================================================

Source of truth: shared/schema.ts
ORM: Drizzle ORM with PostgreSQL
All IDs are UUID strings generated by gen_random_uuid() unless noted.
All timestamps default to now() unless noted.

This section documents every table, every column, what it means, and how
tables relate to each other. Read this before writing any new query or
migration.

────────────────────────────────────────────────────────────────────────────────
TABLE: users
Purpose: One row per registered player account.
────────────────────────────────────────────────────────────────────────────────

  id                    UUID PK         Unique player identifier.
  username              text UNIQUE     Display name (3–20 chars, alphanumeric + underscore).
  email                 text UNIQUE     Login email address.
  password              text            Bcrypt-hashed password. Never stored plain.
  profile_image         text NULL       URL of the player's avatar image.
  coins                 integer         Current coin balance. Never goes negative.
  is_admin              boolean         TRUE grants access to all /api/admin/* routes
                                        and renders the admin panel on the client.
  is_banned             boolean         TRUE blocks login.
  active_pet_id         varchar NULL    FK → user_inventory.id (the pet currently
                                        displayed/used in battle).
  last_username_change  timestamp NULL  Rate-limits username changes.
  last_profile_pic_change timestamp NULL Rate-limits avatar changes.
  password_reset_token  text NULL       One-time token for password reset flow.
  password_reset_expires timestamp NULL Expiry for the reset token.
  market_extra_slots    integer         Extra market listing slots purchased (default 0).
  welcome_v2_sent       boolean         Tracks whether the welcome reward bundle was
                                        already delivered (prevents double-send).
  created_at            timestamp       Account creation time.

RELATIONSHIPS:

  users.id ← coin_purchases.user_id
  users.active_pet_id → user_inventory.id

────────────────────────────────────────────────────────────────────────────────
TABLE: shop_items
Purpose: Master catalogue of every item in the game — pets, accessories,
         fishing gear, fish, potions, power-ups, and special items.
         This is NOT per-player; it is the global item definition table.
────────────────────────────────────────────────────────────────────────────────

  id                    UUID PK
  name                  text            Display name.
  price                 integer         Shop purchase price in coins.
  type                  text            Item category. Valid values:
                                          "pet"       — hatching egg / pet
                                          "power_up"  — stat booster consumable
                                          "item"      — general consumable/resource
                                          "special"   — special-effect item

  When a player casts, the server queries pond_fish for the current location,
  builds a weighted pool using each fish's star_rarity, applies pole/bait

  the player encounters.

SECTION 13 — RARITY / STAR SYSTEM
================================================================================

Rarity is a 1–5 integer used across multiple tables. Its meaning depends
on the item type:

PETS (shop_items.rarity):
  1★ — Common.   Power-up slots per level = 3  × level.
  2★ — Uncommon. Power-up slots per level = 4  × level.
  3★ — Rare.     Power-up slots per level = 5  × level.
  4★ — Epic.     Power-up slots per level = 6  × level.
  5★ — Legendary. Power-up slots per level = 7 × level.

FISH (shop_items.star_rarity — separate field):
  Base catch pool weights: 1★:60  2★:24  3★:10  4★:4  5★:2
  Higher-star fish have shorter nibble windows and fewer nibble chances.
  5★ fish: 1.1 s window, 1 chance.
  1★ fish: 2.5 s window, 3 chances.

ACCESSORIES / ITEMS (shop_items.rarity):
  Higher rarity generally indicates better stat boosts; used for display only
  (coloured stars/borders in the UI). No gameplay formula beyond visuals.

ENEMIES / DROPS:
  Rarity is not a field on enemies; rareness is controlled by drop_rate (%)
  in enemy_drops. Lower drop_rate = rarer drop.

SECTION 14 — HOW FISH FLOW THROUGH THE DATABASE
================================================================================

This traces the full lifecycle of a fish from admin creation to player catch.

1. ADMIN CREATES THE FISH
   POST /api/admin/shop  { type: "fishing", fishingType: "fish", star_rarity: N, ... }
   → Row created in shop_items.

2. ADMIN UPLOADS VISUAL PARTS
   POST /api/admin/fish-parts/:fishItemId  { partType: "Body", image: ... }
   → Row created in fish_template_parts linked to the shop_items.id.
   PATCH /api/admin/fish-parts/:partId  { posX, posY, width, height, zIndex }
   → Positions updated so the minigame knows how to render the fish.

3. ADMIN ASSIGNS FISH TO A POND
   POST /api/admin/location/:locationId/assign-fish/:shopItemId
   → Row created in pond_fish linking location_id + shop_item_id.

4. PLAYER CASTS AT THAT LOCATION
   POST /api/fishing/cast  { locationId, poleInventoryId, baitInventoryId }
   Server queries pond_fish for the location, builds weighted pool, selects fish.
   Returns fish data + fish_template_parts so client can render it.
   Decrements user_inventory.pole_uses_left for the pole.

5. PLAYER COMPLETES THE MINIGAME
   POST /api/fishing/catch  { shopItemId, performanceScore, locationId }
   Server validates performance score, performs final catch roll.
   On success:
     → Row inserted into player_fish_inventory (userId, shopItemId, inAquarium=false).
     → If first time catching this species: row inserted into player_fish_catch_log.
     → If bait was equipped: user_inventory.quantity decremented for bait.

6. PLAYER SELLS OR AQUARIUMS THE FISH
   SELL: DELETE from player_fish_inventory, add coins to users.coins.
   AQUARIUM: PATCH player_fish_inventory.in_aquarium = true.

7. FISH BOOK (COLLECTION)
   GET /api/fishing/catch-log  → returns player_fish_catch_log rows.
   Frontend compares against all available fish (shop_items WHERE fishingType="fish")
   to show which species have been caught and which remain undiscovered.

16.1 — FISHING MINIGAME MECHANICS
================================================================================

────────────────────────────────────────────────────────────────────────────────
PHASE 1: WAITING (after cast)
────────────────────────────────────────────────────────────────────────────────

  Wait time = 1500 ms + random(0 to 2500 ms)
  Range: 1.5 s – 4.0 s

  During this window the server selects the fish from the pond's weighted pool
  (see Section 16.5). The selected fish's rarity determines all subsequent
  difficulty values.

────────────────────────────────────────────────────────────────────────────────
PHASE 2: NIBBLE
────────────────────────────────────────────────────────────────────────────────

  Per-rarity nibble configuration (arrays indexed 0 = 1★ … 4 = 5★):

  Constant                  1★     2★     3★     4★     5★
  ─────────────────────── ────── ────── ────── ────── ──────
  Max nibble chances         3      3      2      2      1
  Window per nibble (ms)   2500   2200   1800   1400   1100

  Source: NIBBLE_MAX_BY_RARITY / NIBBLE_TIMEOUT_BY_RARITY arrays, FishingPage.tsx line 84–85.

  HOW IT WORKS:
  - The first nibble alert fires immediately when the waiting phase ends.
  - If the player misses, the next nibble fires after the window expires.
  - The player taps/clicks anywhere on screen to hook the fish.
  - If ALL chances are exhausted without a tap, the fish escapes and phase
    returns to IDLE (pole durability IS still consumed).

────────────────────────────────────────────────────────────────────────────────
PHASE 3: REELING — BASE RATES PER RARITY
────────────────────────────────────────────────────────────────────────────────

  All rates are per second, scaled each frame by delta-time (dt).
  dt is capped at 2/60 s to prevent spiralling if the tab loses focus.

  Constant                    1★      2★      3★      4★      5★
  ────────────────────────  ──────  ──────  ──────  ──────  ──────
  Starting catch progress    0.08    0.06    0.05    0.03    0.02
  Base reel rate (holding)   0.50    0.42    0.32    0.24    0.17   /s
  Base tension rise (hold)   0.55    0.75    1.05    1.20    1.55   /s
  Tension fall (released)    2.80    2.30    1.80    1.35    0.95   /s
  Progress drain (released)  0.030   0.060   0.100   0.150   0.200  /s
  Escape fill (released)     0.040   0.070   0.110   0.200   0.270  /s
  Escape drain (sweet spot)  0.080   0.060   0.040   0.025   0.010  /s
  Min fish state dur (s)     3.5     3.0     2.5     2.0     1.5
  Max fish state dur (s)     6.0     5.0     4.0     3.0     2.5

  Source: startProgress, baseReelRates, baseTensionRise, etc. arrays,
          FishingPage.tsx lines 418–455.

────────────────────────────────────────────────────────────────────────────────
16.2 — TENSION BAR (SWEET SPOT) MECHANICS
────────────────────────────────────────────────────────────────────────────────

  Tension (T) runs 0.0 – 1.0 (0% – 100%).
  Sweet spot boundaries per rarity (fractions):

  Zone               1★            2★            3★            4★            5★
  ───────────────  ──────────    ──────────    ──────────    ──────────    ──────────
  Low boundary     0.25          0.28          0.30          0.33          0.35
  High boundary    0.75          0.72          0.70          0.67          0.65

  Source: sweetLow / sweetHigh arrays, FishingPage.tsx lines 441–442.

  SWEET SPOT PROGRESS MULTIPLIER (sweetMult):
  ┌─────────────────────────────────────┬──────────┬───────────────────────────────┐
  │ Tension zone                        │ sweetMult│ Effect                        │
  ├─────────────────────────────────────┼──────────┼───────────────────────────────┤
  │ 0 → sweetLow  (too loose)           │   0.5×   │ Half progress, escape grows   │
  │ sweetLow → sweetHigh  (sweet spot)  │   1.0×   │ Full progress, escape drains  │
  │ sweetHigh → 0.90  (risky)           │   1.2×   │ Bonus progress, danger zone   │
  │ 0.90 → 1.0  (critical)              │   0.4×   │ Nearly zero progress, snap    │
  └─────────────────────────────────────┴──────────┴───────────────────────────────┘

  LOSS CONDITIONS (instant):
    Snap:   tension >= 1.0  → snap effect plays (600 ms), then phase → MISSED
    Escape: escapeMeter >= 1.0 → phase → MISSED immediately

  WIN CONDITION:
    catchProgress >= 1.0 → client sends performanceScore = 100 to server

  GRACE PERIOD:
    1200 ms after the nibble tap, the escape meter does NOT fill and catch
    progress does NOT drain. Gives the player time to find the hold button.
    Source: GRACE_MS = 1200, FishingPage.tsx line 504.

────────────────────────────────────────────────────────────────────────────────
16.3 — FISH BEHAVIOR STATES
────────────────────────────────────────────────────────────────────────────────

  Fish cycle through four states. State is picked randomly at intervals
  using the weights below. Each state multiplies the base rates:

  State    | Tension Rise | Reel Progress | Escape Fill | Notes
  ─────────┼──────────────┼───────────────┼─────────────┼──────────────────────
  calm     |     1.0×     |     1.0×      |     1.0×    | Baseline
  resist   |     1.5×     |     0.7×      |     1.4×    | Fish fighting back
  tired    |     0.5×     |     1.5×      |     0.6×    | Player's advantage
  surge    |     2.2×     |     0.5×      |     2.0×    | Must release to survive

  Source: STATE_MODS, FishingPage.tsx lines 497–502.

  STATE PROBABILITY WEIGHTS [calm, resist, tired, surge]:

  Rarity  | Calm | Resist | Tired | Surge
  ────────┼──────┼────────┼───────┼──────
  1★      |  65  |   10   |   20  |   5
  2★      |  55  |   20   |   15  |  10
  3★      |  40  |   30   |   15  |  15
  4★      |  25  |   35   |   15  |  25
  5★      |  15  |   40   |   12  |  33

  Source: stateWeights array, FishingPage.tsx lines 445–451.

────────────────────────────────────────────────────────────────────────────────
16.4 — EQUIPMENT MODIFIER FORMULAS
────────────────────────────────────────────────────────────────────────────────

  FISHING POLE (tension slowdown):
    Only applies to 3★, 4★, 5★ fish.
    poleSlowdownMult = 1 + (poleSlowdownPercent / 100)
    Effective tension rise = baseTensionRise[idx] × poleSlowdownMult
    NOTE: Higher poleSlowdownPercent = MORE tension → harder. Use negative
          values (like -20 → mult 0.80) to make poles helpful.
    Source: FishingPage.tsx lines 461–467.

  BAIT (catch boost):
    Effective reel rate = baseReelRate[idx] × (1 + baitCatchBoost / 100)
    Example: baitCatchBoost = 25 → reel rate × 1.25
    Source: FishingPage.tsx line 466.

  BAIT (rarity boost):
    Applied server-side to the pond weight pool.
    The targeted star tier's weight is multiplied by (1 + rarityBoostPercent / 100).

────────────────────────────────────────────────────────────────────────────────
16.5 — SERVER-SIDE CATCH VALIDATION
────────────────────────────────────────────────────────────────────────────────

  POND SELECTION (weighted random):
    Base weights by star rarity: 1★:60  2★:24  3★:10  4★:4  5★:2
    Rare boost from pole (rareCatchBoostPercent) is added to the 4★ and 5★ weights.
    Bait rarity boost (rarityBoostPercent) is added to the targeted star tier weight.
    The selected rarity group is chosen first; then a random fish from that group.

  PERFORMANCE SCORE:
    Client sends performanceScore (0–100) when catchProgress reaches 1.0.
    A perfect reel = 100. Server formula:

    if score == 100 → catch guaranteed
    else            → catch chance = 0.4 + (score / 100) × 0.5

    Range: score 0 → 40% chance; score 50 → 65% chance; score 99 → 89.5% chance.
    Source: server/routes.ts line ~3508.

  POLE DURABILITY:
    poleUsesLeft is decremented on EVERY CAST regardless of outcome (hit, miss, escape).
    When poleUsesLeft reaches 0, the pole is broken and cannot be used.

  BAIT CONSUMPTION:
    One unit of bait is consumed only on a SUCCESSFUL CATCH.

================================================================================
16.6 — BATTLE / SLASH MECHANICS
================================================================================

────────────────────────────────────────────────────────────────────────────────
SWIPE DETECTION
────────────────────────────────────────────────────────────────────────────────

  Minimum swipe path length to register a hit: 4 (% units across arena).
  Swipe path buffer: last 7 points kept.
  Minimum movement threshold to add a new point: 0.5 (% units).
  Hit detection: segment-to-point distance (segmentPointDist).

  Hitbox radius:
    Normal enemy: 17  (% of arena width/height)
    Boss enemy:   24

  Multi-hit guard: hitEnemiesRef (Set) resets on each new swipe start,
  preventing the same enemy from being hit twice in one continuous motion.
  Source: BattleArena.tsx lines 465–588.

────────────────────────────────────────────────────────────────────────────────
PLAYER DAMAGE TO ENEMY
────────────────────────────────────────────────────────────────────────────────

  Step 1 — Raw damage:
    rawDmg = petAtk - floor(enemyDef × 0.3)

  Step 2 — Random variance:
    variance = floor(random() × 10) - 4   (range: -4 to +5)

  Step 3 — Combo multiplier (hits within 1400 ms window):
    comboMult = 1 + min(comboCount × 0.12, 0.6)

    Combo  | Multiplier
    ───────┼───────────
    1      | 1.00×
    2      | 1.12×
    3      | 1.24×
    4      | 1.36×
    5+     | 1.60× (cap)

  Step 4 — Critical hit:
    Crit chance = 15%
    critMult = 1.9× on crit, 1.0× otherwise

  Step 5 — Counter multiplier (active after a successful parry):
    counterMult = 2.0× (active), 1.0× (inactive)

  FINAL FORMULA:
    damage = max(1, floor((rawDmg + variance) × comboMult × critMult × counterMult))

  Source: BattleArena.tsx lines 489–495.

────────────────────────────────────────────────────────────────────────────────
ENEMY DAMAGE TO PLAYER (FAILED PARRY)
────────────────────────────────────────────────────────────────────────────────

  DEF reduction formula:
    defReduction = def / (def + 60)

    DEF value  | Damage reduction
    ───────────┼─────────────────
    20         | 25.0%
    40         | 40.0%
    60         | 50.0%
    90         | 60.0%
    120        | 66.7%
    180        | 75.0%

  Raw damage:
    rawDmg = enemyAtk × (1 - defReduction)

  Variance:
    + random() × 8 - 4   (range: -4 to +4)

  Final formula:
    damage = max(1, floor(rawDmg + variance))

  Source: BattleArena.tsx line 381–383.

────────────────────────────────────────────────────────────────────────────────
SUCCESSFUL PARRY (BLOCK) — BLEED-THROUGH DAMAGE
────────────────────────────────────────────────────────────────────────────────

  A perfect parry is NOT zero damage. A small amount bleeds through:

  blockAbsorb = min(0.95, def / (def + 30))

    DEF value  | Block absorption | Bleed-through %
    ───────────┼──────────────────┼────────────────
    30         | 50.0%            | 50.0%
    60         | 66.7%            | 33.3%
    90         | 75.0%            | 25.0%
    180        | 85.7%            | 14.3%
    300+       | 95.0% (cap)      | 5.0% (minimum)

  bleedDmg = max(0, floor(enemyAtk × (1 - blockAbsorb) × 0.5))
  (Half the unabsorbed damage comes through.)

  Source: BattleArena.tsx lines 611–613.

────────────────────────────────────────────────────────────────────────────────
PARRY / COUNTER TIMING
────────────────────────────────────────────────────────────────────────────────

  Parry window opens: 320 ms after charge starts
  Charge durations:
    Normal enemy    : 1350 ms  (chargeDurationRef default)
    Boss            : 1100 ms
    Boss in Rage    :  980 ms  (set when boss HP drops to ≤50%)
  Parry window: from 320 ms until charge duration completes.

  Effective parry reaction window:
    Normal enemy: 1350 - 320 = 1030 ms to react
    Boss:         1100 - 320 =  780 ms to react
    Boss Rage:     980 - 320 =  660 ms to react

  Counter state (after successful parry):
    Duration: 3000 ms OR until 2 hits land (whichever comes first)
    Hits granted: 2
    Damage multiplier: 2.0×

  Source: BattleArena.tsx lines 363–370, 427–431, 627–639.

────────────────────────────────────────────────────────────────────────────────
ENEMY ATTACK TIMING (CHARGE INTERVALS)
────────────────────────────────────────────────────────────────────────────────

  difficulty: a random float 0.3 – 1.0, set fresh for each wave.
  Initial charge delay after battle start: 3500 ms (giving player time to react).

  Post-attack cooldown (next charge timer):
    Normal enemy (not boss):
      base = 2600 + random() × 2200 - difficulty × 900
      clamped to minimum 1100 ms
      Range (diff=0.3): ~1700 – 3900 ms
      Range (diff=1.0): ~1700 – 3900 ms (floor applies often at high diff)

    Boss (normal phase):
      base = 3000 + random() × 1500
      Range: 3000 – 4500 ms

    Boss (Rage phase):
      base = 1600 + random() × 1000
      Range: 1600 – 2600 ms

  After a SUCCESSFUL PARRY (enemy recovery):
    Normal:  2600 - difficulty × 400 ms
    Boss:    2200 - difficulty × 400 ms

  Source: BattleArena.tsx lines 418–422, 642–644.

────────────────────────────────────────────────────────────────────────────────
BOSS RAGE
────────────────────────────────────────────────────────────────────────────────

  Triggers when: isBoss = true AND enemyHp <= enemyMaxHp × 0.5
  Once triggered, it never resets within a wave.
  Effect: chargeDurationRef → 980 ms (from 1100 ms)
          Screen gets a red tint.
          Enemy movement becomes more erratic.

  Source: BattleArena.tsx lines 426–431.

16.10 — ENEMY SCALING FORMULAS
================================================================================

  Enemies are scaled to the player's active pet at encounter start.
  Source: server/routes.ts — /api/explore/:locationId/encounter

  ENEMY LEVEL:
    Normal:  petLevel + random(0 to 2)
    Boss:    petLevel + random(0 to 5)

  ENEMY HP:
    base = max(200, floor(petHp × 2 × bossHpMult × waveScaling))
    bossHpMult:   4.0 for bosses, 1.0 for normal
    waveScaling:  1 + (waveIndex × 0.15)
    (Wave 0 = ×1.0, wave 1 = ×1.15, wave 2 = ×1.30, etc.)

  ENEMY ATK / DEF:
    base = max(statFloor, floor(petStat × (2/3) × bossStatMult × waveScaling))
    bossStatMult: 2.5 for bosses, 1.0 for normal

  DIFFICULTY MODIFIER:
    Per-wave random float: 0.3 – 1.0
    Used only to modulate attack frequency (see 16.6 charge intervals).

16.12 — DROP RATE SYSTEM
================================================================================

  Drop rates are stored in enemy_drops.drop_rate as an integer 1–100.
  Each drop on the loot table is rolled INDEPENDENTLY:

    roll = random() × 100
    if roll < drop_rate → item drops

  CONSEQUENCE: Multiple items can drop from a single enemy kill.
  A drop_rate of 100 = guaranteed drop every kill.
  A drop_rate of 10 = 10% chance per kill.
  Drop rates are capped between 1 and 100 on creation (admin enforced).

  Source: server/routes.ts line ~2924.

16.13 — COIN REWARD SYSTEM
================================================================================

  BASE COINS:
    Fixed per enemy, set in location_enemies.coin_reward.
    Awarded immediately on enemy defeat before any drop rolls.

  DUPLICATE PET COMPENSATION:
    If a pet drop item is already owned by the player (an owned instance exists
    with the same shop_item_id), instead of giving a second copy, the server
    awards a coin equivalent based on the item's shop price.
    Source: server/routes.ts lines ~2625–2636.

  FISH SELLING:
    Fish sell for their shop_items.price in coins.
    Fish can be sold from the Fish Inventory screen or at a Fish Barrel
    in the world scene.

================================================================================
16.14 — HATCHING SYSTEM
================================================================================

  TIMER:
    hatchTime is stored on shop_items in HOURS.
    Hatching starts when the player initiates it, setting hatchStartedAt.
    Progress = (now - hatchStartedAt) / (hatchTime × 3600 × 1000)

  HATCH ACCELERATOR ITEMS:
    specialType = "hatch_time", specialAmount = hours to skip.
    The server shifts hatchStartedAt backward by (specialAmount × 3600 × 1000 ms).
    Multiple accelerators can be stacked — each one shifts further back.
    If the accumulated shift exceeds the total hatch time, the egg hatches instantly.

  HATCH CHECK ENDPOINT:
    GET /api/pet/:inventoryId/hatch-check
    Compares (now - hatchStartedAt) >= (hatchTime hours in ms).
    On success: sets isHatched = true on user_inventory.

  Source: server/routes.ts lines ~963, ~1195–1204.

  On EQUIP:
    petAtk  += accessory.atkBoost  (if > 0)
    petDef  += accessory.defBoost  (if > 0)
    petHealth += accessory.healthBoost  (if > 0)

  On UNEQUIP:
    petAtk  -= accessory.atkBoost
    petDef  -= accessory.defBoost
    petHealth -= accessory.healthBoost

  Attempting to equip a 4th without unequipping one returns an error.
  Current HP is adjusted proportionally if maxHp changes from a health accessory.

16.17 — BALANCE GUIDANCE FOR FUTURE EDITS
================================================================================

  When editing any mechanic, follow these rules:

  FISHING BALANCE:
  - To make a rarity EASIER:  lower baseTensionRise, raise tensionFalls,
    lower baseEscapeRate, or widen the sweetLow/sweetHigh range.
  - To make a rarity HARDER:  reverse the above, or add more surge weight.
  - Do NOT change per-rarity arrays independently — they must remain
    internally consistent across all 5 rarity tiers.
  - Pole slowdown is multiplicative ON TOP of baseTensionRise, not additive.
    A slowdownPct of 30 means tension rises 30% MORE, not less. This is
    currently counter-intuitive — see note in source code.

  BATTLE BALANCE:
  - Increasing pet damage: raise statBoostAmount on power-up items, or adjust
    the enemy DEF reduction formula (def × 0.3 scaling).
  - Increasing difficulty: raise difficultyRef ceiling, or shorten charge
    durations. Do NOT remove the parry window without replacing the mechanic.
  - Counter hits (2) and duration (3000 ms) are intentionally low — increasing
    them significantly can trivialise boss fights.
  - Crit chance (15%) should remain below 20% to avoid feeling random/unfair.

  LEVELING BALANCE:
  - The XP formula (100 + level × 30 + level² × 5) creates steep late-game
    scaling. Any change to this formula invalidates all existing pet XP.
    Plan a data migration if changing it.
  - Power-up slots per level (rarity + 2) is a soft cap, not a hard one —
    players can always level up to unlock more slots.

  DROP BALANCE:
  - drop_rate values should stay between 2–50 for most items.
  - drop_rate = 100 should only be used for guaranteed quest/tutorial drops.
  - Because each drop is independent, having many drops at 25% each means
    a player will often receive multiple items per kill.
